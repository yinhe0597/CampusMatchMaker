## 1. 总则与适用范围

### 1.1 目的
统一团队开发节奏与协作标准，降低沟通成本，保证代码质量，让"分层解耦、逻辑复用"的架构理念在工程层面真正落地。

### 1.2 适用范围
适用于本项目所有参与开发、测试、产品、运维的成员。

### 1.3 核心原则
- **小步快跑**：优先交付可验证的最小功能单元。
- **单点打透**：MVP 阶段聚焦"时间投票"主线，不擅自扩散。
- **配置优先**：能用配置实现的，不写死代码。
- **质量内建**：测试、评审、文档是开发的一部分，不是收尾工作。

## 2. 团队角色与职责

| 角色 | 职责 |
|------|------|
| 产品负责人（PO） | 定义需求优先级、维护需求池、验收功能 |
| 技术负责人（TL） | 把控架构、审核架构红线、主持技术评审 |
| 开发工程师 | 任务开发、编写单元测试、提交评审 |
| 测试工程师 | 编写测试用例、执行测试、把控发布质量门禁 |
| 评审人（Reviewer） | 审核代码，至少 1 名非作者成员 |

> 小团队可一人多角，但**作者本人不可作为自己代码的唯一评审人**。

## 3. 开发流程总览

```
需求提出 → 需求评审 → 拆解任务 → 排期认领
   ↓
创建分支 → 开发 + 自测 → 提交 PR
   ↓
代码评审 → 修改 → 通过
   ↓
合并到测试分支 → 测试验证
   ↓
合并主干 → 发布 → 验收 → 关闭任务
```

每个功能都必须走完整流程，**禁止跳过评审或测试直接合并主干**。

## 4. 需求与任务管理规范

### 4.1 需求来源
所有需求统一进入"需求池"，由产品负责人按优先级排序。

### 4.2 需求评审
新需求开发前必须评审，明确：
- **价值**：解决什么问题，服务哪个场景。
- **范围边界**：明确"做什么"与"不做什么"。
- **架构归属**：属于原子能力层 / 工作流层 / 内容层？是否可用配置实现？
- **隐私影响**：是否涉及用户敏感数据？是否符合"默认最小暴露"？

### 4.3 任务拆解
- 单个任务粒度建议控制在 **1-3 天可完成**。
- 任务标题清晰，包含：模块 + 动作 + 目标。
  - 例：`[课表] 实现班级公共课表自动继承逻辑`
- 任务需注明：所属阶段、优先级、负责人、验收标准。

### 4.4 任务状态流转
```
待办 → 进行中 → 待评审 → 测试中 → 已完成
```

## 5. 分支管理规范

### 5.1 长期分支

| 分支 | 用途 | 保护级别 |
|------|------|---------|
| `main` | 生产环境，永远可发布 | 受保护，禁止直接 push |
| `develop` | 集成开发分支，最新可测试代码 | 受保护，需 PR 合入 |

### 5.2 临时分支

| 分支类型 | 命名规范 | 来源 → 去向 |
|---------|---------|------------|
| 功能分支 | `feature/模块-简述` | `develop` → `develop` |
| 修复分支 | `fix/模块-简述` | `develop` → `develop` |
| 紧急修复 | `hotfix/简述` | `main` → `main` + `develop` |

**命名示例：**
- `feature/vote-recommend-timeslot`
- `fix/timetable-inherit-error`

### 5.3 分支规则
- 每个任务对应一个独立分支，**禁止在 `develop` 直接开发**。
- 功能完成后通过 PR 合并，**禁止直接 merge 不经评审**。
- 分支合并后及时删除，保持仓库整洁。
- 长期分支需定期同步，避免大冲突。

## 6. 编码规范

### 6.1 通用原则
- **命名见名知意**，禁止 `a`、`tmp1`、`data2` 等无意义命名。
- **单一职责**：一个函数只做一件事，函数体不宜过长。
- **避免魔法数字**：用具名常量替代散落的数字。
- **注释解释"为什么"**，而非"做了什么"（代码本身应自解释）。

### 6.2 分层约束（重要）
严格遵守三层架构的依赖方向：
- **原子能力层**：只提供纯能力，**禁止包含任何具体业务逻辑**（如"投票""排班"等概念）。
- **工作流层**：通过配置组合原子能力，**禁止绕过引擎自己写匹配逻辑**。
- **内容层**：**禁止重复实现**已有的内容管理与标签能力。

> 违反分层约束的代码，评审一律打回。

### 6.3 配置优先原则
- 新增工作流场景，**优先通过配置模板实现**。
- 如需修改原子引擎，必须经技术负责人评审，确认该改动是"通用能力"而非"特定场景需求"。

## 7. 提交规范

### 7.1 提交信息格式
```
<类型>(<范围>): <简短描述>
```

### 7.2 类型说明

| 类型 | 含义 |
|------|------|
| feat | 新功能 |
| fix | 修复缺陷 |
| docs | 文档变更 |
| style | 格式调整（不影响逻辑） |
| refactor | 重构（非新功能非修复） |
| test | 测试相关 |
| chore | 构建/工具/配置变更 |

### 7.3 示例
```
feat(vote): 增加共同空闲时段自动推荐
fix(timetable): 修复同班继承时选修课误覆盖问题
docs(workflow): 补充工作流配置模板说明
```

### 7.4 提交要求
- **一次提交只做一件事**，禁止"大杂烩"提交。
- 提交信息使用中文或英文均可，但团队需统一。
- 禁止提交：临时文件、密钥配置、注释掉的废弃代码。

## 8. 代码评审规范

### 8.1 提交评审（PR）要求
PR 描述需包含：
- **做了什么**：本次改动的内容。
- **为什么**：对应的需求/任务。
- **如何验证**：测试方式与结果。
- **影响范围**：是否触及原子能力层、是否影响其他功能。

### 8.2 评审检查清单
评审人需逐项确认：
- [ ] 是否符合三层架构与依赖方向？
- [ ] 原子能力层是否被混入了业务逻辑？
- [ ] 新场景是否优先用了配置而非硬编码？
- [ ] 是否符合隐私规范（默认最小暴露、只输出结果）？
- [ ] 命名、注释、可读性是否达标？
- [ ] 是否有对应的测试？
- [ ] 是否有安全隐患（敏感数据暴露、权限缺失）？

### 8.3 评审纪律
- 每个 PR **至少 1 名非作者评审通过**方可合并。
- 触及**原子能力层或隐私逻辑**的 PR，必须由技术负责人评审。
- 评审意见对事不对人，作者需逐条回应。
- 评审未通过禁止合并。

## 9. 测试规范

### 9.1 测试分层

| 层级 | 测试重点 |
|------|---------|
| 原子能力层 | **必须有单元测试**，覆盖时段计算、标签匹配的各种边界情况 |
| 工作流层 | 集成测试，验证引擎组合与流程正确性 |
| 内容层 | 功能测试，验证内容展示与标签定向 |

### 9.2 重点测试用例（针对本项目）
- **空闲时段计算**：无人有空、全员有空、部分重叠、跨天、临界时间点。
- **课表继承**：首次录入、自动继承、选修补充、协作纠错。
- **标签匹配**：无匹配、完全匹配、加权排序正确性。
- **隐私边界**：确认投票结果是否泄露了个人日程明细。

### 9.3 测试要求
- 原子能力层改动**必须补充/更新单元测试**。
- 功能合并到 `develop` 前需通过自测。
- 发布前由测试工程师执行回归测试。

## 10. 发布与部署规范

### 10.1 发布流程
```
develop 测试通过 → 合并 main → 打版本标签 → 部署 → 验收 → 公告
```

### 10.2 版本号规范
采用语义化版本：`主版本.次版本.修订号`
- **主版本**：架构级重大变更。
- **次版本**：新增功能（如上线"找搭子"）。
- **修订号**：缺陷修复、小优化。

例：`v1.2.0` 表示在 1.x 架构下新增了第 2 个功能。

### 10.3 发布质量门禁
满足以下条件方可发布：
- [ ] 所有相关测试通过。
- [ ] 代码已评审合并。
- [ ] 隐私检查清单确认通过。
- [ ] 关键功能完成回归测试。
- [ ] 文档已同步更新。

### 10.4 紧急修复
线上严重缺陷走 `hotfix` 流程，修复后**必须同时合并回 `develop`**，避免下次发布回退问题。

## 11. 文档规范

### 11.1 必须维护的文档
- **架构文档**：三层架构说明与依赖关系。
- **工作流配置文档**：每个工作流模板的配置说明。
- **引擎能力文档**：原子能力的输入/输出/边界说明。
- **变更记录**：每个版本的功能变更与修复记录。

### 11.2 文档原则
- 代码与文档同步更新，**文档过期视为缺陷**。
- 新增工作流模板时，必须同步补充配置说明。

## 12. 缺陷处理流程

```
发现缺陷 → 提交缺陷单 → 定级 → 分配 → 修复 → 验证 → 关闭
```

### 缺陷分级

| 级别 | 定义 | 处理时效 |
|------|------|---------|
| 致命 | 核心功能不可用 / 数据泄露 | 立即处理 |
| 严重 | 主要功能异常 | 当日处理 |
| 一般 | 次要功能问题 | 排期处理 |
| 轻微 | 体验/文案问题 | 择机处理 |

> **涉及隐私/数据泄露的缺陷一律定为"致命"级**，优先处理。

## 13. 架构红线（本项目专属铁律）

为守护"分层解耦、逻辑复用"的核心理念，以
下以红线**任何人不得逾越**，违反者代码一律打回并需重构：

### 红线一：原子能力层的纯粹性
- ❌ 原子能力层**禁止出现任何业务概念**（"投票""排班""饭搭子"等词不应出现在引擎代码中）。
- ❌ 引擎**禁止直接读写具体业务数据表**，只接受标准化输入、返回标准化输出。
- ✅ 引擎应当像一个数学函数：相同输入永远得到相同输出，不关心调用者是谁。

### 红线二：工作流层禁止"绕道"
- ❌ 禁止在工作流层自己重写一套时段计算或标签匹配逻辑。
- ❌ 禁止因为"赶时间"就把匹配逻辑硬编码进具体功能。
- ✅ 新场景一律先问："能否用现有引擎 + 配置实现？" 不能，再讨论是否扩展引擎。

### 红线三：隐私不可妥协
- ❌ 任何功能**禁止对外输出个人原始日程明细**。
- ❌ 禁止默认公开用户标签、课表等敏感信息。
- ✅ 对外只输出聚合/计算结果，遵循"默认最小暴露"。

### 红线四：MVP 阶段的克制
- ❌ MVP 阶段禁止擅自开发"时间投票"主线之外的功能。
- ❌ 禁止为了"架构完整"而过度配置化、过度抽象。
- ✅ 先验证单点，再释放架构能力。新功能必须经产品负责人排期。

### 红线五：数据信任链
- ❌ 禁止让单个用户的错误录入无审查地影响整个班级。
- ✅ 班级公共课表必须有协作纠错机制兜底。

## 14. 协作沟通规范

### 14.1 日常同步
- **每日简短同步**：各成员说明"昨天做了什么、今天做什么、有无阻塞"。
- **阻塞优先**：遇到卡点立即在群内提出，不憋到当天结束。

### 14.2 技术决策
- 涉及**架构、引擎、隐私**的决策，必须经技术负责人确认，并记录决策理由（便于日后回溯）。
- 重大分歧无法达成一致时，由技术负责人拍板，但需记录不同意见。

### 14.3 沟通原则
- **对事不对人**：评审、讨论聚焦问题本身。
- **结论留痕**：重要讨论结果落到文档或任务单，不停留在口头。

## 15. 新成员上手指南

新成员加入时，须按以下顺序熟悉项目：

1. **读产品介绍文档**：理解"在正确的时间找到正确的人"的核心理念。
2. **读架构文档**：理解三层架构与依赖方向。
3. **读本工作流规范**：掌握协作流程与红线。
4. **跑通本地环境**：完成一次完整的"拉分支 → 改代码 → 自测 → 提 PR"流程。
5. **从小任务起步**：第一个任务建议为轻量级缺陷修复，熟悉流程后再接功能开发。

> 上手第一周，所有 PR 由技术负责人重点指导评审。

## 16. 规范的维护与演进

- 本规范**不是一成不变的**，随项目阶段演进而调整。
- 任何成员都可提出规范优化建议，经团队讨论后更新。
- 规范变更需记录版本与变更原因，并通知全体成员。

### 规范演进的阶段提示

| 项目阶段 | 规范侧重点 |
|---------|-----------|
| MVP 阶段 | 强调"单点打透"与"克制"，流程可适当轻量 |
| 扩展阶段 | 强化"配置优先"与"架构红线"，确保复用红利兑现 |
| 成熟阶段 | 强化测试覆盖、发布门禁、文档完整性 |

## 结语

这份工作流规范的目的，**不是用流程束缚团队，而是用清晰的约定守护两件最重要的事**：

1. **守护架构**——让"分层解耦、逻辑复用"不只是 PPT 上的理念，而是每一行代码都遵守的纪律。
2. **守护用户**——让隐私安全与产品克制成为团队的本能，而非事后补救。

记住贯穿全文的那句话：**别爱上自己的架构而忘了用户只想解决一件具体的小事。** 流程是为了让我们更快、更稳地把那件小事做到极致，而不是相反。
