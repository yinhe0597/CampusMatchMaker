## 1. 架构总览

### 1.1 三层架构图

```
┌──────────────────────────────────────────────────────────────┐
│                      内容层（阶段五）                          │
│      失物招领 · 活动预告 · 赞助广告                             │
│      —— 复用内容管理 + 标签定向能力                             │
├──────────────────────────────────────────────────────────────┤
│                      工作流层（可配置）                        │
│      时间投票 · 找搭子 · 值班排班 ...                          │
│      —— 用配置模板组合原子能力，新增功能 = 新增一份配置          │
├──────────────────────────────────────────────────────────────┤
│                      原子能力层（基础引擎）                    │
│      ① 共同空闲时段计算引擎（TimeSlotEngine）                  │
│      ② 标签加权匹配引擎（TagMatchEngine）                      │
│      —— 不关心上层业务，只提供纯数学计算能力                    │
├──────────────────────────────────────────────────────────────┤
│                      数据基础层                                │
│      用户认证 · 课表数据 · 标签画像 · 权限                      │
└──────────────────────────────────────────────────────────────┘
```

### 1.2 依赖方向（严格单向，禁止反向）

```
handler（HTTP接口层）
    ↓ 调用
service（业务编排层）
    ↓ 调用          ↓ 调用
workflow          domain（数据访问层）
    ↓ 调用
engine（原子能力层）
```

**关键约束**：
- `engine` 不依赖任何上层包（handler、service、workflow、domain 均不可 import）
- `workflow` 依赖 `engine`，但禁止依赖 `handler`
- `domain` 不依赖 `workflow` 和 `engine`
- `handler` 只做参数校验、调用 service、组装响应，不含业务逻辑

## 2. 各层职责与边界

### 2.1 原子能力层（engine）

**定位**：系统的心脏，像数学函数一样运作——相同输入永远得到相同输出。

| 特征 | 要求 |
|------|------|
| 纯粹性 | 不含任何业务概念（投票、排班、饭搭子等词禁止出现） |
| 无副作用 | 不读写数据库、不调用外部 API、不依赖全局状态 |
| 标准化 | 只接受标准化输入，返回标准化输出 |
| 可测试 | 必须有高覆盖率单元测试（≥ 90%） |

### 2.2 工作流层（workflow）

**定位**：系统的灵魂，通过配置（而非代码）组合原子能力，"长"出各种功能。

| 特征 | 要求 |
|------|------|
| 配置驱动 | 新场景优先通过配置模板实现 |
| 禁止绕道 | 禁止自己重写时段计算或标签匹配逻辑 |
| 编排者 | 负责定义流程步骤、调用引擎、处理业务状态 |

### 2.3 内容层（content）

**定位**：平台运行的副产物，复用已有的内容管理与标签定向能力。

### 2.4 数据基础层（domain）

**定位**：管理领域模型与数据访问，包含用户、班级、课表等核心领域对象。

### 2.5 接口层（handler）

**定位**：HTTP 入口，职责单一——参数校验 → 调用 service → 组装响应。

## 6. 核心引擎算法设计

### 6.1 共同空闲时段计算引擎（TimeSlotEngine）

#### 算法流程

```
输入：一组用户的日程数据
  ↓
① 输入标准化：将课表条目转为"已占用时段"列表
  格式：{ day_of_week(1-7), start_minutes, end_minutes }
  ↓
② 合并重叠：对每个用户，合并其内部重叠的已占用时段
  ↓
③ 取反得空闲：从全周时间轴中扣除已占用时段，得到每个用户的空闲时段
  时间轴范围：每天 08:00 ~ 22:00（可配置）
  ↓
④ 多人求交：对 N 个用户的空闲时段求交集
  ↓
⑤ 聚合统计：统计每个时段的可参与人数、参与率
  ↓
⑥ 排序输出：按参与率降序排列
  ↓
输出：[{ date, start, end, available_count, total, rate }]
```

#### 数据结构定义

```go
// ===== 输入 =====

// OccupiedSlot 表示一个用户的单个已占用时段
// 注意：不包含课程名、教室等任何业务信息，只有时间
type OccupiedSlot struct {
    DayOfWeek    int // 1=周一 ... 7=周日
    StartMinutes int // 从 00:00 起的分钟数，如 08:00 = 480
    EndMinutes   int // 从 00:00 起的分钟数，如 10:00 = 600
}

// UserSchedule 一个用户的完整日程
type UserSchedule struct {
    UserID string
    Slots  []OccupiedSlot
}

// ===== 输出 =====

// FreeSlotResult 一个推荐时段
type FreeSlotResult struct {
    DayOfWeek      int     // 周几
    StartMinutes   int     // 开始时间（分钟）
    EndMinutes     int     // 结束时间（分钟）
    AvailableCount int     // 有空人数
    TotalCount     int     // 总人数
    Rate           float64 // 参与率 (0.0 ~ 1.0)
}

// ===== 配置 =====

// EngineConfig 引擎运行配置
type EngineConfig struct {
    DayStartMinutes int // 每天有效开始时间，默认 480 (08:00)
    DayEndMinutes   int // 每天有效结束时间，默认 1320 (22:00)
    SlotGranularity int // 时间粒度（分钟），默认 30
}
```

#### 边界条件处理

| 场景 | 处理方式 |
|------|---------|
| 无人有空 | 返回空列表 + 提示"当前无共同空闲时段" |
| 全员有空 | rate = 1.0，排在最前 |
| 部分重叠 | 取交集，只保留所有人都有空的部分 |
| 跨午休时段 | 按 DayStart/DayEnd 配置过滤，不特殊处理 |
| 时间粒度对齐 | 以 SlotGranularity（默认30分钟）为单位对齐 |
| 单人场景 | 直接返回该用户的空闲时段，rate = 1.0 |

#### 隐私保证

- 引擎只接收 `OccupiedSlot`（纯时间数据），**不接收课程名、教室、教师等信息**
- 输出只有聚合统计（人数、参与率），**无法反推任何个人的具体日程**
- 引擎是无状态的纯函数，不持久化任何数据

工作流模板注册后，系统自动在前端菜单中生成对应入口：
1. 后端启动时加载 `templates/` 目录下所有 JSON 配置
2. 提供 `GET /api/v1/workflows` 接口返回可用工作流列表
3. 前端根据返回列表动态渲染左侧菜单

### 7.3 MVP 阶段策略

MVP 阶段仅实现 `time_vote` 一个工作流模板，**不做过度抽象**。配置化机制在阶段三进行反向抽象。

## 8. 隐私架构设计

### 8.1 数据流隐私审计

| 数据流节点 | 隐私级别 | 处理方式 |
|-----------|---------|---------|
| 用户课表原始数据 | 高敏感 | 加密存储，API 默认不返回他人课表 |
| 学号 (users.student_id) | 高敏感 | **SHA-256 哈希存储**，确定性查找，不可逆推 |
| 学号 (student_auth.student_id) | 中敏感 | 明文存储，仅用于认证审核流程 |
| 引擎输入（OccupiedSlot） | 中敏感 | 只含时间，不含课程信息 |
| 引擎输出（FreeSlotResult） | 低敏感 | 只有聚合统计，可安全展示 |
| 投票结果 | 低敏感 | 只展示参与率，不展示个人投票明细 |
| 用户标签画像 | 高敏感 | 匹配前只展示模糊画像 |

### 8.2 隐私中间件

```go
// PrivacyFilter 响应隐私过滤中间件
// 功能：自动从 HTTP 响应中移除敏感字段
// 规则：
//   - /api/v1/timetables/* 响应中，若请求者非本人，移除课表详情
//   - /api/v1/polls/*/results 响应中，移除个人投票明细
//   - 所有响应中，移除手机号、学号等 PII 字段（除非是本人请求）
```

### 8.3 授权模型

| 原则 | 实现 |
|------|------|
| 默认最小暴露 | 所有隐私字段默认关闭，需主动开启 |
| 场景化授权 | 投票场景只暴露聚合结果，不暴露个人课表 |
| 可撤销 | 用户可随时关闭已授权的数据共享 |
| 只输出结果 | 对外只展示"计算结果"，不展示"原始数据" |

## 9. 架构红线与代码保障

### 9.1 红线一：原子能力层纯粹性

**规则**：`engine/` 包禁止 import 任何业务包。

**保障措施**：
- CI 加入 import 依赖检查脚本，自动检测 `internal/engine/` 是否引入了 `domain/`、`workflow/`、`handler/` 等包
- Code Review 检查清单必检项

### 9.2 红线二：工作流层禁止绕道

**规则**：`workflow/` 禁止自己实现时段计算或标签匹配。

**保障措施**：
- `workflow/` 包禁止直接操作数据库（不 import `domain/` 的 repository）
- 所有计算必须通过调用 `engine/` 包完成

### 9.3 红线三：隐私不可妥协

**规则**：任何功能禁止对外输出个人原始日程明细。

**保障措施**：
- `handler/middleware/privacy.go` 统一过滤响应
- `engine` 的输入输出数据结构中不含个人信息字段

### 9.4 红线四：MVP 阶段克制

**规则**：MVP 只实现时间投票。

**保障措施**：
- `workflow/` 下只实现 `poll/` 目录
- 产品排期严格控制

### 9.5 红线五：数据信任链

**规则**：单用户错误录入不可无审查地影响全班。

**保障措施**：
- `timetable_corrections` 表 + 纠错审批状态机
- 公共课表修改需经过审核，不允许单用户直接修改

## 10. 非功能性设计

### 10.1 性能目标

| 场景 | 目标 |
|------|------|
| 空闲时段计算（50人班级） | P99 < 200ms |
| API 常规请求 | P99 < 100ms |
| 课表继承（首次加载） | < 500ms |

### 10.2 可扩展性

- 后端无状态设计，可水平扩展
- 数据库读写分离预留（GORM 支持）
- Redis 缓存层可按需引入

### 10.3 可观测性

- 结构化 JSON 日志（Zap）
- 请求追踪 ID（X-Request-ID）
- 日志中敏感信息自动脱敏（学号、手机号）

### 10.4 安全

- 密码使用 bcrypt 哈希存储
- JWT Token 设置合理过期时间
- API 限流（可选，阶段二后引入）
- SQL 注入防护（GORM 参数化查询）
- CORS 白名单配置
