校园协作平台 · 产品介绍与开发说明
---
第一部分：产品介绍
1.1 一句话定位
一个让校园师生"在正确的时间，找到正确的人"的智能协作平台。
我们把繁杂的校园协作需求（约时间、找搭子、排值班）抽象为两种最基础的能力——日程与标签，再用一个通用匹配引擎驱动所有上层应用，让每一个新场景都能被快速、低成本地搭建出来。

核心洞察：这些看似五花八门的需求，本质都是"在合适的时间，匹配到合适的人"。只要把"时间"和"人"做成可复用的底层能力，所有场景都能被统一驱动。
1.3 产品核心理念
逻辑复用至上，分层解耦。
• 原子能力层：最底层，只做两件事——算空闲时段、算标签匹配度。
• 工作流层：用配置（而非写代码）组合原子能力，"长"出投票、找搭子、排班等各种功能。
• 内容层：失物招领、活动预告等，复用已有内容与标签能力，作为平台运行的副产物。
这套架构的价值：新增一个功能，不需要重写匹配逻辑，只需在后台定义一个配置模板，左侧菜单就自动出现新功能。
1.4 首发核心场景（MVP）
班级 / 社团内，基于自动继承的课表，一键发起活动时间投票，系统智能推荐共同空闲时段。
这是我们的"尖刀场景"：高频、刚需、依赖熟人封闭群体（不依赖陌生人密度，易冷启动），且能集中验证产品的三个核心假设。
1.5 我们的差异化优势
1. 课表智能继承：第一个同学结构化录入班级课表，后续同班同学自动继承，无需重复填写。
2. 空闲时段自动计算：发起投票时，系统直接推荐"全班 80% 都有空的黄金时段"，而非让用户手动勾选——这是对微信接龙的降维打击。
3. 隐私默认安全：只输出"计算结果"，不暴露"个人日程明细"，让用户放心。
4. 架构可无限扩展：一套底层能力，支撑无限协作场景。

│   失物招领 · 活动预告 · 赞助广告            │
│   —— 复用内容管理 + 标签定向能力            │
├─────────────────────────────────────────┤
│   工作流层（可配置）                        │
│   时间投票 · 找搭子 · 值班排班 ...          │
│   —— 用配置模板组合原子能力                 │
├─────────────────────────────────────────┤
│   原子能力层（基础引擎）                     │
│   ① 共同空闲时段计算引擎                     │
│   ② 标签加权匹配引擎                         │
│   —— 不关心上层业务，只提供纯能力            │
├─────────────────────────────────────────┤
│   数据基础层                                │
│   用户认证 · 课表数据 · 标签画像 · 权限      │
└─────────────────────────────────────────┘2.2 原子能力层（系统的心脏）
引擎一：共同空闲时段计算引擎
• 输入：一组用户的日程数据。
• 处理：将每个人的"已占用时段"取并集，再从总时间轴中扣除，得到空闲时段；对多人空闲时段求交集，并按"有空人数比例"排序。
• 输出：推荐时段列表（如"周四 14:00-16:00，可参与率 85%"）。
• 关键原则：只输出聚合结果，绝不输出"谁在什么时间忙/在上什么课"。
引擎二：标签加权匹配引擎
• 输入：需求方的标签条件 + 候选人的属性标签。
• 处理：区分"需求标签"（如辣度=微辣）与"属性标签"（如食堂=一食堂），按权重计算匹配度得分。
• 输出：按匹配度排序的候选人列表。
• 关键原则：匹配前只展示模糊画像，双向确认后才解锁详情。
2.3 工作流层（系统的灵魂）
工作流 = 一份配置模板，定义"调用哪些引擎、按什么顺序、需要哪些参与方、如何呈现结果"。
示例：
• 时间投票 = 选择参与者 → 调用空闲时段引擎 → 展示推荐时段 → 收集投票 → 确认。
• 找搭子 = 填写需求标签 → 调用标签匹配引擎 → 展示候选 → 发起邀约 → 双向确认。
• 值班排班 = 选择人员 → 调用空闲引擎 → 按规则分配 → 确认。
核心价值：新增功能 = 新增一份配置，无需改动底层引擎代码。
2.4 数据基础层
• 用户认证：学号实名认证。
• 课表数据：班级公共课表（自动继承）+ 个人选修课表（个人补充）。
• 标签画像：兴趣、技能、偏好等标签。
• 权限系统：默认最小暴露，场景化授权。

第三部分：开发说明
3.1 开发原则
1. 底层先行：先把两个原子引擎做稳，它们是一切的基础。
2. 单点打透：MVP 阶段只做"时间投票"一条线，做到极致，不铺开。
3. 配置驱动：工作流层从一开始就用配置化思路设计，避免后期重构。
4. 隐私内建：隐私不是后期补丁，而是设计阶段就遵循"默认最小暴露"。
3.2 开发阶段规划
阶段一：地基（数据 + 认证 + 课表）
目标：把"人"和"时间"两种基础数据立起来。
• 学号实名认证流程。
• 课表分层录入系统：
    ◦ 第一个同学结构化录入班级公共课表（填表形式：周几 / 第几节 / 课程名，不是上传图片）。
    ◦ 后续同班同学自动继承公共课表。
    ◦ 个人补充选修课。
    ◦ 协作纠错机制：发现继承的课表有误，可标记修正。
• 空闲时段的自动计算逻辑（基于课表数据）。
本阶段验证：用户是否愿意录课表？继承机制能否真正降低门槛？
阶段二：尖刀功能（时间投票）
目标：跑通第一个完整工作流，验证核心体验。
• 创建投票活动（选择班级/社团成员为参与者）。
• 调用空闲时段引擎，第一屏直接展示推荐时段（核心体验，不可弱化）。
• 成员确认/投票。
• 结果汇总与最终时间确定。
本阶段验证：自动推荐空闲时段，是否真的比微信接龙更爽？
阶段三：架构沉淀（工作流配置化）
目标：把投票功能"反向抽象"为配置模板，验证扩展性。
• 把已跑通的投票功能改造为一份配置模板。
• 设计工作流配置结构（定义参与方、引擎调用顺序、结果呈现）。
• 实现"配置变更后，菜单自动生成新功能"的机制。
阶段四：能力扩展（找搭子 / 排班）
目标：用配置模板"长"出新功能，验证复用红利。
• 完善标签画像与标签匹配引擎。
• 通过新增配置模板，搭建"找搭子""值班排班"功能。
• 完善双向确认与隐私授权机制。
阶段五：内容层与管理闭环
目标：补全生态。
• 首页广场：失物招领、活动预告、赞助广告（复用内容+标签能力）。
• 管理端：举报审核、内容管理。
• 数据分析：活跃度、功能使用情况等。
3.3 隐私设计规范（贯穿全程）
原则
具体要求

默认最小暴露
默认任何人都看不到他人具体课表

只输出结果
投票只显示"可参与率"，不显示谁忙、忙什么

场景化授权
找搭子匹配前只展示模糊画像，确认后才解锁

开关克制
用合理的默认设置替代大量隐私开关，降低用户决策负担

3.5 成功标准（MVP 验收指标）
• 录入转化：班级中第一个用户完成课表录入的比例。
• 继承有效性：后续同学自动继承课表的成功率。
• 体验优势：用户在"自动推荐时段"上的使用率与满意度，对比接龙。
• 复购：班级/社团第二次、第三次发起投票的比例（验证是否真刚需）。
---
结语
这个产品的精髓在于：对外，它先是一个把"约时间"做到极致的好用工具；对内，它是一套能支撑无限校园场景的优雅架构。
开发上务必克制——先用"时间投票"这一把尖刀切入市场，验证用户真的愿意录数据、真的会复用，再让底层架构的复用能力逐步释放出来。 别同时铺开所有功能，别爱上架构而忘了用户只想解决一件具体的小事。
